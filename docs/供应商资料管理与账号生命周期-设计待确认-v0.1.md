## 2. 议题一 · 企业资料编辑

### 2.1 字段按"身份强度"分两类

| 类别 | 字段 | 性质 |
|---|---|---|
| **身份字段(强约束)** | `name`(公司名)、`country_code`(注册国)、`registration_no`(企业凭证号) | 决定"这家公司是谁",改 = 重审 |
| **运营字段(弱约束)** | 联系电话、联系邮箱、公司简介、logo、业务范围、营业额等(**当前模型没有这些列**) | 维护用,改不重审 |

### 2.2 状态机 × 字段类型 编辑权限矩阵(建议)

| 状态 | 身份字段 | 运营字段 | 备注 |
|---|---|---|---|
| **DRAFT**(刚注册,未提交审核) | ✅ owner 可改 | ✅ owner 可改 | 入驻前的自由编辑期 |
| **UNDER_REVIEW**(已提交,等审) | ❌ 锁 | ❌ 锁 | 改了审核基线就乱 |
| **APPROVED**(审核通过) | ❌ 锁;真要改 = "申请变更"另起单 | ✅ owner 可改,不触发重审 | 长期数据治理 |
| **REJECTED**(被拒) | ✅ owner 改完重新提交 | ✅ owner 可改 | 修正后再审 |
| **DISABLED**(被平台禁用) | ❌ 全锁 | ❌ 全锁 | 退出态 |

### 2.3 MVP 范围切分(由小到大)

| Layer | 内容 | 工作量(粗估) | 价值密度 |
|---|---|---|---|
| **A** | 企业资料**只读**展示(把 DRAFT 公司信息显示在 profile 页) | 半天 | ⭐ |
| **B** | A + **运营字段**编辑(先建模:`SupplierOrganization` 加 phone / 简介 / 业务范围等新列 + migration) | 2-3 天 | ⭐⭐ |
| **C** | B + **DRAFT 时身份字段**也能改(公司名 / registration_no) | +0.5 天 | ⭐⭐⭐ |
| **D** | C + **提交审核动作** + **审核流程**(OPERATOR 后台审,`DRAFT → UNDER_REVIEW → APPROVED/REJECTED`) | 1 周 | ⭐⭐⭐⭐ |
| **E** | D + **APPROVED 后申请变更**(变更单走二次审核) | 数天 | ⭐⭐ |

### 2.4 待用户拍板的问题

| # | 问题 | 候选 | 我的倾向 |
|---|---|---|---|
| Q1 | MVP 切到哪一层 | A / B / C / D / E | A + C(不加新列、不做审核) |
| Q2 | "联系电话/邮箱" 归属 | 个人(`User.phone/email`)/ 企业(新增 `SupplierOrg.phone/contact_email`)/ 两个都要 | 只用个人字段,等后续真有需求再拆 |
| Q3 | 运营字段范围(若做 Layer B+) | logo / 简介 / 业务范围 / 营业额 / 营业执照扫描件 / ... | 待你列具体清单 |
| Q4 | 本期做不做审核流程 | 做 / 推迟 | **推迟**(等业务真有审核诉求再做,且要先有 OPERATOR 后台) |
| Q5 | APPROVED 后申请变更 | 做 / 推迟 | **推迟**(跟 Q4 联动) |

### 2.5 我的整体推荐(待你拍)

**Layer A + C**:
- 把 placeholder 替换成真实展示(name / country / registration_no / status)
- DRAFT 状态下 owner 可编辑这 3 个身份字段
- 不加新列、不做审核、不做申请变更
- 工作量 ~1 天,把"profile 页是空的"这个明显坑填上,后续审核流程独立工单

## 3. 议题二 · 账号注销与 Owner 注销

### 3.1 业界惯例对比

| 场景 | 是否提供自助注销 | 原因 |
|---|---|---|
| **C 端**(微信、抖音、淘宝、滴滴) | ✅ 提供(常带冷静期、清算条件) | 个人账号,法律强制(GDPR / PIPL / CCPA) |
| **B 端 SaaS**(钉钉、飞书、Slack、Notion) | ❌ 基本不提供 / 或要管理员审批 | 账号是企业资产,业务数据交织;走"管理员禁用" |
| **政企供应链 / 金融 / 医疗** | ❌ 几乎都不提供 | 合规审计要求长期数据保留 |

**我们项目语境**:央企海外 EPC 供应链 = 标准 B 端 + 合规敏感场景 → 主流是**不做自助注销**,走 OPERATOR 后台禁用。

### 3.2 SUPPLIER Owner 注销的四种策略(若真做)

当前模型:`SupplierMember.is_owner=True` 标识 owner,**单 owner**(MVP 阶段大多数 SUPPLIER 只有 1 个 owner)。

| 策略 | 做法 | 利弊 |
|---|---|---|
| **A. 禁止注销 owner** | owner 必须先把所有权转交给某 member 才能走注销 | 最稳;但 MVP 阶段大多组织只有 1 owner,实际等于"不能注销" |
| **B. 整组销户** | owner 注销 = 组织同步标记 `DISABLED`,SupplierOrg 退出平台 | 适合"1 人公司退出";若已有业务记录(报价、订单)处理复杂 |
| **C. 静默接管** | 系统自动选一个 member 顶上(按加入时间最早) | 用户感知差、法律风险(未经新 owner 同意)→ **不推荐** |
| **D. 平台托管** | owner 注销 = 组织进入"无主"状态,OPERATOR 后台介入处理(可重新指派 / 可解散) | 兜底方案,人工介入,实施简单 |

### 3.3 数据策略

| 策略 | 实现 | 适用场景 |
|---|---|---|
| **硬删** | `DELETE FROM users WHERE id=X` | GDPR 被遗忘权强制要求 → 我们不适合 |
| **软删** | `User.status='DISABLED'` + `disabled_at` 时间戳 | **主流**;我们的 `UserStatus.DISABLED` 已有,加个时间戳即可 |
| **匿名化** | 软删 + 个人信息脱敏(`name='已注销用户'`、`email='deleted+{id}@platform.local'`、`phone=NULL`)+ 保留业务关联 | GDPR 友好;MVP 暂无出海合规需求,可延后 |

### 3.4 待用户拍板的问题

| # | 问题 | 候选 | 我的倾向 |
|---|---|---|---|
| Q6 | 用户自助注销做不做 | 做 / 不做 | **不做**(B 端惯例 + MVP 合规优先) |
| Q7 | 若不做,代偿方案 | 个人资料页加引导文字 + 邮件链接 / 完全不提 | 加引导文字 |
| Q8 | OPERATOR 后台禁用员工功能 | 已有 / 待新增 / 待确认 | 待你或我下次摸代码确认 |
| Q9 | 若做注销 - Owner 策略 | A / B / D(放弃 C) | D(平台托管),最稳 |
| Q10 | 数据策略 | 硬删 / 软删 / 匿名化 | 软删 + 加 `disabled_at` |

### 3.5 我的整体推荐(待你拍)

**MVP 阶段:完全不做用户自助注销**。

**理由**:
1. B 端业务账号本来就不该用户自助删,主流做法都不提供
2. Owner 注销链路一旦做就要顺带"所有权转交 / 整组销户 / 平台托管",一口气拉很长
3. 真有诉求 → OPERATOR 后台 `status=DISABLED` 一行解决
4. 跟"个人资料编辑"的范围讨论一致 —— 先把基础生命周期跑顺,注销留到合规真正需要(出海多国上线)再做

**代偿**:个人资料页加一条"如需停用账号,请联系平台运营 [邮箱]"的引导,把诉求引到人工通道,零代码。

**推荐演进顺序**:
1. 确认 / 补齐 OPERATOR 后台禁用员工功能
2. 业务跑一段时间观察真实自助注销诉求
3. 出现真实需求 / 出海合规要求 → 再做"用户自助申请注销"工单流(D 策略 + 软删)
