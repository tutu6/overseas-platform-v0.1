| 字段 | 内容 |

## 一、背景与上下文

### 1.1 业务场景

信用评估模块需要对以下文档进行自动化识别和理解,辅助生成供应商画像评分:

1. **营业执照**(海外属地营业执照 + 国内营业执照)
2. **资质证书**(SASO / CIDB / SNI / SONCAP 等海外准入证书)
3. **检测报告**(第三方质量检测报告)
4. **财务报表**(后续场景,V1.0+)

每个场景不仅需要**字段提取**,还需要**理解和判断**:

- 证书是否在有效期内
- 证书覆盖的品类与目标国是否匹配
- 证书签发机构是否权威
- 营业执照是否疑似造假
- 关键字段交叉验证是否一致

### 1.2 决策驱动因素

| 因素 | 说明 | 权重 |
|---|---|---|
| **成本** | MVP 阶段控成本,技术负责人明确要求"用国内相对低成本的大模型" | 高 |
| **合规** | 央企客户场景,数据不出境是硬要求 | 高 |
| **采购可行性** | 公司已有阿里云合作,内部采购流程顺畅 | 高 |
| **场景适配性** | 需要同时支持字段提取 + 理解判断 | 高 |
| **识别精度** | 海外证书 / 多语种文档识别准确率达标 | 高 |
| **代码可演进** | 支持未来切换模型,避免锁定 | 中 |
| **生态一致性** | 跟未来 AI Agent 工具箱方向一致 | 中 |

### 1.3 约束条件

- 不允许使用境外 LLM 服务(Claude / GPT 等)作为 MVP 阶段主力,以满足成本和合规要求
- 数据必须在境内处理
- MVP 阶段预算有限,单次 OCR 调用成本应控制在合理范围

---

## 二、决策

### 2.1 决策内容

**MVP 阶段采用通义千问 VL 系列(qwen-vl-max)作为 OCR / 文档理解的默认模型。**

具体落地方案:

1. **主力模型**:`qwen-vl-max`(阿里云 DashScope)
2. **接口抽象**:在后端封装 `OCRService` 抽象层,所有 OCR 调用走该接口
3. **降级方案**:遇到识别准确率不足或服务不可用时,标记为"待人工核验",由运营处理
4. **数据留痕**:所有 OCR 调用的原始返回写入 `ocr_extraction_log` 表,保留至少 1 年

### 2.2 范围与边界

**本决策适用于**:

- 信用评估模块下的所有 OCR 场景
- 供应商入驻流程中的证照识别
- 未来 AI Agent 工具箱中"证书审查 Agent"的底层 OCR 能力

**本决策不适用于**:

- 风控驾驶舱中的图谱可视化(不涉及 OCR)
- 大批量结构化文档处理(V1.0+ 量上来后另行决策,见"未来演进")

## 三、备选方案对比

| 方案 | 优势 | 劣势 | 是否选用 |
|---|---|---|---|
| **通义千问 qwen-vl-max** | 阿里云生态、国内合规、价格可控、中文识别强、API 兼容 OpenAI 格式 | 海外多语种证书识别效果需实测验证、对长 prompt 遵循度略弱于 Claude | ✅ **选用** |
| Claude Vision(Anthropic) | 多语种理解强、海外文档结构识别好、prompt 工程友好 | 数据出境合规风险、国内访问需代理、成本相对高、采购流程复杂 | ❌ 不符合"国内低成本"要求 |
| GPT-4o Vision | 多模态能力强 | 同 Claude,合规与访问问题更突出 | ❌ 同上 |
| DeepSeek-VL2 | 价格便宜、开源系 | 商用 API 稳定性不足、生态不如阿里云完善 | ❌ MVP 阶段稳定性优先 |
| 智谱 GLM-4V | 国内合规、多模态能力中等 | 价格高于通义千问、海外场景训练数据相对少 | ❌ 性价比不如通义 |
| 阿里云文档智能(读光) | 价格极低(几分钱/张)、速度快、结构化输出准确 | 仅做提取不做判断、对非标证书识别差 | ⚠️ V1.0+ 大规模场景备选 |
| 阿里云传统 OCR | 最便宜、最快 | 只输出纯文字,无结构化和理解能力 | ❌ 不满足判断场景需求 |
| 通义千问 qwen-vl-plus | 比 max 便宜 | 复杂判断任务能力不足 | ⚠️ 简单字段提取场景备选 |
| 通义千问 qwen-vl-ocr | OCR 专用、识别精度高 | 弱化了理解判断能力 | ⚠️ 纯提取场景备选 |

## 四、后果

### 4.1 正面影响

- ✅ 成本可控,符合 MVP 阶段预算要求
- ✅ 数据不出境,合规风险低
- ✅ 阿里云生态,与对象存储 OSS、数据库等服务集成方便
- ✅ DashScope API 兼容 OpenAI 格式,代码迁移成本低
- ✅ 公司内部采购流程顺畅,启动速度快

### 4.2 负面影响 / 风险

- ⚠️ 海外阿拉伯文 / 印尼文证书识别效果需要实测验证,可能存在准确率不达标的风险
  - **缓解措施**:MVP 启动前用真实样本做基准测试(见"实施计划")
- ⚠️ 通义千问对长 prompt 遵循度略弱,复杂判断任务需要精细的 prompt 工程
  - **缓解措施**:采用 few-shot 示例 + JSON Schema 约束 + 多步拆解
- ⚠️ 锁定阿里云生态,未来切换成本可能上升
  - **缓解措施**:`OCRService` 抽象接口设计,业务代码与具体模型解耦
- ⚠️ DashScope QPS 限制相对严格
  - **缓解措施**:异步队列调用 + 重试机制 + 限流保护

### 4.3 技术债务标记

- 海外多语种识别效果未经验证,需在 MVP 启动前完成基准测试
- 复杂判断场景的 prompt 工程能力需要持续优化
- 未来批量场景的成本可能不优,需要在 V1.0+ 评估混合方案

## 五、实施计划

### 5.1 阶段一:验证(MVP 启动前)

1. 申请阿里云 DashScope 账号 + qwen-vl-max 调用权限
2. 准备真实样本:海外营业执照 5 张、海外证书 5 张(SASO/CIDB/SNI 各 1-2 张)、国内营业执照 3 张
3. 编写基准测试脚本,跑一遍样本,记录:
   - 字段提取准确率(≥90% 达标)
   - 判断任务准确率(有效期判断、签发机构识别等)
   - 平均响应时间
   - 失败率
4. **验收标准**:综合准确率 ≥85% 进入下一阶段;<85% 触发备选方案评估

### 5.2 阶段二:实施(MVP 开发期)

1. 实现 `OCRService` 抽象接口(`backend/app/services/ocr/`)
2. 实现 `QwenVLOCRService` 具体类
3. 设计并创建 `ocr_extraction_log` 表
4. 实现异步队列调用 + 重试机制
5. 编写营业执照识别 prompt + 证书识别 prompt(带 few-shot 示例)
6. 接入供应商入驻流程

### 5.3 阶段三:监控与优化(MVP 上线后)

1. 监控指标:调用量、成功率、平均响应时间、单次成本
2. 持续收集识别失败的样本,反哺 prompt 优化
3. 月度成本审查,评估是否需要切换混合方案

---

## 六、未来演进

本决策仅适用于 MVP 阶段。以下条件触发时,需评估新方案并写新 ADR:

| 触发条件 | 评估方向 |
|---|---|
| 月 OCR 调用量超过 5000 次 | 评估"读光 + qwen-vl-max"混合方案 |
| 海外证书识别准确率持续低于 85% | 评估切换其他模型或人工兜底比例 |
| 阿里云价格政策重大变动 | 重新评估性价比 |
| 国家合规政策变化允许境外模型 | 评估是否引入 Claude / GPT 作为备选 |
| 出现新一代国产多模态模型(明显优于 qwen) | 评估切换 |
