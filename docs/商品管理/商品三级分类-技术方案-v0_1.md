# 商品三级分类 · 技术方案 v0.1

> 适用版本:MVP 第二轮(分类底座)

---

## 0. 一分钟速览

- **做什么**:给平台建一张商品三级分类表,源数据来自中建三局给的 Excel,本轮只做"分类底座",不做绑定/接入。
- **怎么做**:单表自关联(L1→L2→L3),分类编码用 `01.001.001` 2-3-3 数字格式,Excel 一键导入脚本,两个查询 API,前端三级联动组件。
- **本轮不做什么**:供应商主营品类绑定、商品-分类绑定、商城品类导航、分类后台管理界面。这些都留给后续轮次。
- **对 PM 的影响**:本轮没有可视化产出供产品验收,只交付"底座 + demo 页"。供应商入驻接入分类、商城按品类筛选等用户可见功能,会在后续轮次的 PRD 里再约定。

## 1. 为什么要做这件事

平台从一开始就需要一个**统一的商品分类口径**,几个核心场景都依赖它:

| 场景 | 依赖方式 |
|---|---|
| 供应商入驻 | 选择"主营品类"(后续轮次接入) |
| 商城商品上架 | 每个 SKU 必须挂在某个三级分类下(后续轮次接入) |
| 商城前台导航 | 按品类浏览(后续轮次接入) |
| 采购清单 / 询价 | 按品类聚合(后续轮次接入) |

中建三局已经给了一份《三局产品三级分类(整合合同分类)》Excel,我们这一轮就是把这份口径**数据化、查询化、组件化**——后续业务功能用的时候,直接调 API、直接用组件,不再重复造分类轮子。

## 3. 数据库设计

### 3.1 表结构

| 字段 | 类型 | 用途 |
|---|---|---|
| `id` | Integer 自增 PK | 技术主键,ORM 内部用 |
| `code` | String(16) UNIQUE NOT NULL | **业务主键**,关联用,**永久不变**(见 §3.3) |
| `name_zh` | String(128) NOT NULL | 中文名,可改 |
| `name_en` | String(128) NULL | 留空,等翻译 Agent 那一轮再补 |
| `level` | Integer NOT NULL | 1 / 2 / 3 |
| `parent_code` | String(16) NULL,FK→`categories.code` | 父节点的 code,L1 为 NULL |
| `sort_order` | Integer NOT NULL,默认 0 | 同级排序 |
| `is_active` | Boolean NOT NULL,默认 true | 启用/停用(不是软删,真删走 Excel 重灌 + flag) |
| `created_at` / `updated_at` | DateTime NOT NULL,UTC | 标准时间戳 |

**索引**:`UNIQUE(code)`、`INDEX(parent_code, level)`、`INDEX(level, is_active, sort_order)`。

### 3.2 code 编码规则

格式:**`XX.XXX.XXX`,纯数字点分,2-3-3**——对齐 UNSPSC / 京东 MRO 团标的业界事实标准。

```
01                      围护幕墙              ← L1 两位
  01.001                  玻璃幕墙            ← L2 两位+三位
    01.001.001              明框玻璃幕墙      ← L3 两位+三位+三位
    01.001.002              隐框玻璃幕墙
  01.002                  金属幕墙
02                      机电安装
```

**容量**:L1 容量 99 / L2 每个一级下 999 / L3 每个二级下 999。建工 EPC 实际数据 L1 约 10-30 个、L2 约 30-80 个、L3 约几十到一百多,**3-10 倍冗余**,撞墙概率接近零。

**为什么不用语义短码**(如 `ELEC_CABLE_POWER`):语义短码会和中文名绑死——一旦"电力电缆"改名"动力电缆",code 改还是不改都是坑。纯数字 code 没有语义、不承担命名,改名时永远不需要动 code。

### 3.3 code 永久不变契约(★ 关键设计)

**`code` 一旦分配,永久不变。** 这是本表对外提供"业务主键"的安全承诺,所有引用方据此假设设计。

**什么操作不会让 code 变**(放心做):

- 改中文名 / 英文名 / 排序 / 启用状态
- Excel 中调换顺序
- Excel 中新增、删除其他节点(走 §4.3 "沿用已有 code"算法)

**什么会让 code "看起来变"**:节点物理迁移到不同父级下。处理方式:**旧节点停用 + 新位置分配新 code**,关联表的数据通过运维脚本逐条迁移,**永不**就地改 code。

**为什么要立这个契约**:将来供应商主营品类、商品-分类绑定等表都会用 `category_code` 做外键。如果 code 会变,所有引用方都得跟着改,代价巨大。把"永不变"承诺写死,引用方可以放心。

## 4. Excel 导入设计

### 4.1 Excel 格式

源文件:`data/三局产品三级分类(整合合同分类)20260516.xlsx`

| 列 | 内容 | 说明 |
|---|---|---|
| A | 一级分类 | 必填 |
| B | 二级分类 | 必填 |
| C | 三级分类 | 一格可多值,用 `、,,;;\n\r` 分隔 |

表头模糊匹配("一级分类 / 一级 / Level1" 都认),匹配不上直接 fail-fast 并打印实际表头。

### 4.2 文件治理

- Excel 入 Git(MVP 阶段数据非敏感,所有人复现方便)
- 文件必须放在项目根 `data/` 目录下,**脚本拒绝读 `data/` 外的路径**(防误指)
- `data/` 下可同时放多份历史 xlsx;脚本默认取 mtime 最新;歧义时打印列表提示用 `--file` 指定

### 4.3 "沿用已有 code"算法(兑现 §3.3 契约)

```
对 Excel 每个节点(L1→L2→L3 顺序):
  查 DB 中是否已存在(以 name_zh + parent_code 为匹配键)
  ├─ 已存在 → 沿用旧 code,只 upsert name_zh / sort_order / is_active
  └─ 不存在 → 取"同父下当前最大序号 + 1"分配新 code,insert
```

**关键不变量**:

- 匹配键是 `(name_zh, parent_code)`,**不是 Excel 行号**——Excel 重排不影响 code
- 新节点取空号,**不抢占**老 code
- Excel 删除的节点,DB 里**默认保留**(`--deactivate-missing` 才会停用),**永不物理删**

### 4.4 脚本使用

```bash
# 默认:扫 data/ 取最新 xlsx
python scripts/import_categories.py

# dry-run:只打印 新增/更新/保留不动/将停用 差异,不写库
python scripts/import_categories.py --dry-run

# 显式指定文件(路径必须在 data/ 下,脚本会校验)
python scripts/import_categories.py --file ../data/xxx.xlsx

# 停用 Excel 中已删除的分类(谨慎用)
python scripts/import_categories.py --deactivate-missing
```

**幂等**:同一 Excel 重复跑,记录数 / code / 字段值都不变。

### 4.5 灌数时机

**本轮:本地开发者手动跑脚本一次**。不挂应用启动、不挂 alembic 迁移、不挂部署流水线。

> **部署侧"如何让生产环境自动灌数"是单独的议题**,等本轮业务功能完成后另开任务讨论。本 PRD 不预设、不预留接口、不在代码里搭框架,保持业务和部署解耦。

## 5. 后端 API

两个公开接口,无需登录、无需权限点。

### 5.1 路由

| Method | Path | 用途 | 典型场景 |
|---|---|---|---|
| GET | `/api/v1/categories` | 扁平数组 | 单层查询、调试 |
| GET | `/api/v1/categories/tree` | 三层嵌套 | 前端三级联动一次性拉全树 |

### 5.2 查询参数

`/api/v1/categories`:`level`(可选,只查某层) / `parent_code`(可选,只查某父下的子级) / `is_active`(默认 true)

`/api/v1/categories/tree`:`is_active`(默认 true)

### 5.3 响应示例

**扁平**:

```json
{"code": 0, "message": "ok", "data": [
  {"id": 1, "code": "01", "name_zh": "围护幕墙", "level": 1, "parent_code": null, "sort_order": 0}
]}
```

**树形**:

```json
{"code": 0, "message": "ok", "data": [
  {"code": "01", "name_zh": "围护幕墙", "level": 1, "children": [
    {"code": "01.001", "name_zh": "玻璃幕墙", "level": 2, "children": [
      {"code": "01.001.001", "name_zh": "明框玻璃幕墙", "level": 3, "children": []}
    ]}
  ]}
]}
```

### 5.4 性能

全表预计 < 1000 条,不加缓存,每次内存建树。后续若变慢再加进程内 TTL 缓存(**禁止引入 Redis**,CLAUDE.md 红线)。

## 6. 前端组件

### 6.1 位置

```
frontend/src/components/category/CategoryCascader.tsx   # 三级联动组件
frontend/src/lib/api/categories.ts                       # API 封装
frontend/src/hooks/useCategoryTree.ts                    # SWR 拉树
```

### 6.2 组件签名

```ts
type SelectedCategory = {
  level1Code: string | null;   // "01"
  level2Code: string | null;   // "01.005"
  level3Code: string | null;   // "01.005.012"  ← 业务关联用这个(叶子节点)
};

<CategoryCascader
  value={selected}
  onChange={(v) => setSelected(v)}
  required?: boolean;
  disabled?: boolean;
/>
```

### 6.3 行为

- 首次渲染:SWR 拉 tree,加载中 disabled
- 选 L1 → L2/L3 自动清空,L2 下拉重算
- 选 L2 → L3 自动清空,L3 下拉重算
- 上级未选时下级 disabled,提示"请先选上级"
- 数据为空显示"暂无数据,请联系管理员"

### 6.4 本轮接入

只在临时 demo 页 `test/category` 用一下,**不接入真实业务表单**。供应商入驻 Step 2、商品发布表单、商城品类导航这些都留给后续轮次。

## 8. 关键决策汇总

| # | 决策点 | 结论 | 理由 |
|---|---|---|---|
| D1 | code 编码规则 | 纯数字 `01.001.001`(2-3-3 点分) | 对齐 UNSPSC/MRO 团标;不绑语义,改名不影响;肉眼可读 |
| D2 | Excel 三级单元格多值是否拆分 | 拆,用 `、,,;;\n\r` | 业务数据天然长这样 |
| D3 | Excel 是否入 Git | 入 Git(`data/` 目录) | MVP 数据非敏感、量小,所有人复现方便 |
| D4 | Excel 删除分类怎么处理 | 默认保留,`--deactivate-missing` 才停用,永不物理删 | 最安全,误删代价低 |
| D5 | 本轮是否做供应商主营品类绑定 | 不做,单独一轮 | 避免和已上线的供应商注册 v1.4 打架 |
| D6 | 灌数时机:自动 vs 手动 | 本轮只做本地手动 CLI;部署侧自动灌数另议 | 业务和部署解耦;一次性操作,后续可补 |

## 9. 已知风险

| 风险 | 影响 | 应对 |
|---|---|---|
| Excel 表头与预期不符 | 导入失败 | 模糊匹配 + fail-fast 报错并打印实际表头 |
| 同名节点歧义(同 L2 下两个同名 L3) | 视为同一节点合并 | 导入日志打 warn,提醒人工核对 |
| 二级分类未来撞 999 上限 | 极端情况 | 业务上拆分一级,不扩位(扩位破坏 code 不变契约) |
| 生产部署后 categories 表为空 | 用户访问异常 | **本轮范围外**,留待业务完成后另开"部署自动灌数"任务 |
