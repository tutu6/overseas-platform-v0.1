> **版本** v0.1 · **受众** 领导 / PM / 后端开发
> **关联** RBAC 设计同步说明 · MVP 业务流程共识

## 1. 方案概述

### 1.1 目标

为央企海外工程供应链平台提供**认证与授权基础设施**,作为后续业务功能(项目、采购、询价、订单、履约)的统一底座。核心能力:用户注册登录、基于角色的访问控制(RBAC)、数据归属隔离、操作审计、多组织模型。

### 1.2 设计原则

**够用就好,留好扩展口子**——MVP 按标准 RBAC0 实现,不做角色继承、不做缓存、不做策略引擎;但 schema 留足扩展空间,未来加新角色 / 新组织 / 新权限点都不改表结构。

### 1.3 不在范围

业务功能页面、邮件 / 找回密码 / OAuth / 2FA、消息推送、PWA。

## 3. 数据模型(9 张表)

| 类别 | 表 | 用途 |
|---|---|---|
| **RBAC 核心(5)** | users / roles / permissions / user_roles / role_permissions | 标准 RBAC0 五件套 |
| **组织模型(4)** | buyer_organizations / buyer_members / supplier_organizations / supplier_members | 采购方与供应商各一对 |
| **审计(1)** | audit_logs | 操作审计,带 trace_id 全链路 |

**关键设计要点**:

- **users**:email / username / phone 三种登录凭证,均唯一约束,bcrypt 哈希,`must_change_password` 标记强制首次改密
- **roles**:`code` 唯一标识,`scope` 字段预留(MVP 全 GLOBAL,未来组织内角色用 BUYER_ORG / SUPPLIER_ORG)
- **permissions**:`<resource>:<action>` 格式(如 `rfq:approve`),`module` 用于分组
- **buyer/supplier_organizations**:两类组织独立建表。**采购方**含企业名 + 统一社会信用代码(唯一约束),**供应商**含企业名 + 营业执照号(唯一约束) + 入驻状态机
- **buyer/supplier_members**:中间表,字段 `user_id / org_id / is_owner / created_at`。**不含 business_role**——"用户在组织内的角色"未来通过 user_roles + roles.scope 表达
- **audit_logs**:trace_id / user / resource / action / method / path / ip / status / extra,4 个索引

**角色清单(MVP 阶段 4 个)**:

| code | scope | 用户类型 | 业务定位 |
|---|---|---|---|
| BUYER | GLOBAL | external | 采购方:能做所有采购操作(发起询价、管理项目、订单、履约,含自审批) |
| SUPPLIER | GLOBAL | external | 供应商:能做所有供应商操作(档案、上架、报价、履约) |
| OPERATOR | GLOBAL | internal | 平台运营:业务审核、监控,不触系统配置 |
| ADMIN | GLOBAL | internal | 系统超管:用户/角色/系统配置,不触业务数据 |

MVP 阶段权限点约 33 个。

📌 **是否升级为 6 角色**(引入组织内职级 PURCHASER + OVERSEAS_LEADER):见 §8 待团队拍板。

## 4. 核心模块设计

### 4.1 RBAC 启动同步机制

**问题**:角色-权限关联存数据库 seed 改起来麻烦;写代码常量未来加管理后台又要重构。

**方案**(已决):配置在代码文件(`rbac/permissions_config.py`,跟 Git 走),启动时同步到 DB。

```
应用启动 → 读 permissions_config.py 的 ROLE_PERMISSIONS
       → 与 DB role_permissions 表比对
       → 配置多出的 INSERT,DB 多出的 DELETE
       → 输出 [+N / -N / 不变 N] 统计
```

**dry_run 模式**:`PERMISSION_SYNC_MODE=dry_run` 时只输出差异不执行,用于上线前预检查。同步只触 permissions / role_permissions / roles 三张表,不动 user_roles / 业务数据,保证幂等。

### 4.2 权限校验三级机制

```
用户操作 → 后端 API 守卫(安全底线,必做)
        → 前端路由守卫(无权页面跳转)
        → 前端按钮显隐(UX 体验)
```

后端是安全底线,前端是 UX 优化——前端隐藏可绕过,后端不能。

**后端实现**(FastAPI 依赖注入):

```python
@router.post("/rfq/approve")
async def approve_rfq(
    current: CurrentUser = Depends(require_permission("rfq:approve")),
    db: AsyncSession = Depends(get_db),
):
    # 进入这里说明 current.permissions 已含 "rfq:approve"
    ...
```

`require_permission` 内部:从 JWT 解析 user_id → 实时查 DB 拿 roles + permissions → 校验权限点 → 不通过返 403。

**为何 permissions 不放 JWT**:用户禁用 / 角色变更时,JWT 旧权限会失效。每次实时查 DB 保证权威性,MVP 量级足够。

### 4.3 数据权限:归属字段过滤

权限点回答"能不能做",归属过滤回答"能对哪些数据做":

- **公开池**(已上架商品、已审核供应商、国别准入):无 WHERE 过滤
- **私有池**(项目、询价、订单等):service 层强制按归属字段过滤——采购方按 `buyer_org_id`,供应商按 `supplier_id`

不上策略引擎,简单 if-else 即可。

### 4.4 注册与登录流程

**采购方注册**(完全自助,按信用代码识别企业):

```
POST /auth/register/buyer
{ email, username, phone, password, name,
  company_name, unified_social_credit_code }
  → 校验密码强度(8-32 位 + 1 字母 + 1 数字) + 三类凭证唯一性
  → 按 unified_social_credit_code 查 BuyerOrganization:
      ├─ 不存在 → 创建新 BuyerOrg → 用户成为 owner(is_owner=true)
      └─ 已存在 → 用户加入该 BuyerOrg(is_owner=false)
  → 创建 User + BuyerMember + 赋 BUYER 角色
  → 写审计日志
```

**供应商注册**(完整自助流程):

```
POST /auth/register/supplier
{ ...同上 + company_name, business_license_no }
  → 校验密码 + 唯一性 + 营业执照号唯一
  → 创建 User → 创建 SupplierOrganization(status=DRAFT)
  → 创建 SupplierMember(is_owner=true)
  → 赋 SUPPLIER 角色 → 写审计日志
```

供应商后续走 OPERATOR 审核(业务流程 1):DRAFT → UNDER_REVIEW → APPROVED / REJECTED。

**登录**(支持 email / username / 手机号三选一):

```
POST /auth/login { identifier, password }
  → identifier 自动识别:含 @ 当 email,纯数字当 phone,其他当 username
  → 限流检查(60s 内 5 次失败 → 锁 5 分钟)
  → 校验密码 hash + 账号状态
  → 签发 access_token(JWT,2h) + refresh_token
  → 写审计日志(成功 + 失败都写)
```

`must_change_password=true` 用户登录可成功,但其他业务接口被拦,必须先调改密(中间件 + 前端路由强制)。

## 5. 关键接口

### 5.1 接口分组

| 路径 | 用途 | 主要接口 |
|---|---|---|
| `/api/v1/auth/*` | 认证 | register/buyer · register/supplier · login · logout · me · change-password |
| `/api/v1/admin/users/*` | 用户管理(权限点 `user:manage`) | 创建账号 · 列表 · 禁用 |
| `/api/v1/me/*` | 个人资料 | 改 profile · 改 email(需验密) · 改 phone · 设置 username |
| `/api/v1/test/*` | RBAC 联调 | 4 个测试接口,各限定不同角色访问 |

### 5.2 统一规范

| 维度 | 规范 |
|---|---|
| 响应包装 | `{ code, data, message }` 成功失败统一格式 |
| 错误处理 | 全局异常拦截器,业务异常映射 HTTP 状态码 |
| 时间格式 | ISO 8601 with timezone |
| 分页 | `?page=1&size=20`,响应带 total/page/size |
| Trace ID | 每个请求生成 UUID,贯穿日志与审计 |

## 6. 安全与合规

### 6.1 等保 2.0 三级要求落地

| 要求 | 实现 |
|---|---|
| 身份鉴别 | bcrypt 哈希 + 密码强度规则 + 登录限流 |
| 访问控制 | RBAC + 数据归属过滤 + 业务/系统管理严格分离(OPERATOR vs ADMIN) |
| 安全审计 | 全部敏感操作写 audit_logs,带 trace_id 全链路 |
| 数据完整性 | 数据库唯一约束 + 外键约束 |
| 个人信息保护 | 密码全程哈希存储,日志脱敏 |

### 6.2 关键安全设计

- **密码规则**:8-32 位 + 至少 1 字母 + 1 数字,bcrypt cost=12
- **登录凭证**:email / username / phone 三选一,均唯一约束;手机号格式校验(中国大陆 11 位)
- **JWT 安全**:不带 permissions(只带 user_id),短有效期 + refresh token
- **登录限流**:(identifier, ip) 维度,60s 内 5 次失败锁 5 分钟
- **super admin**:env 注入初始密码,`must_change_password=true`
- **职责分离**:OPERATOR 与 ADMIN 互斥,数据库 + 业务双层约束

### 6.3 审计日志策略

**写**:登录/登出/改密/注册/用户管理/角色变更/业务关键操作(审批、状态流转)/失败的权限校验。**不写**:GET 查询类(避免日志爆炸)。

### 7.1 本地启动

**前置依赖**:Docker 24+ / Docker Compose v2

```bash
git clone <repo-url> && cd overseas-platform
cp .env.example .env                # 生成 JWT_SECRET_KEY:openssl rand -hex 32
                                    # 本地开发设 SEED_DEMO_ACCOUNTS=true

docker compose exec backend python -m app.seed      # 跑 seed
```

## 8. 风险与待定

### 8.1 已知风险

| 风险点 | 缓解措施 |
|---|---|
| 限流单机内存,多实例绕过 | MVP 够用,起量切 Redis |
| seed demo 账号生产隐患 | `SEED_DEMO_ACCOUNTS` 开关默认关闭,部署前 review |
| 权限实时查 DB 高并发瓶颈 | MVP 量级无压力,起量加缓存 + 失效机制 |
| BUYER 角色"自审"业务不真实 | MVP 演示导向,如需职级拆分见 Q28 |
| 采购方完全自助注册无审核 | 生产可能需引入 OPERATOR 审核环节 |

### 8.2 待团队拍板

| 编号 | 议题 | 当前方案 |
|---|---|---|
| Q27 | 采购方注册是否需引入审核环节 | MVP 自助注册;生产可能引入 OPERATOR 审核 |
