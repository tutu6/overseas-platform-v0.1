## 一、定位与边界

### 1.1 定位

第一阶段信用评估模块是"海外工程领域专业版企查查"。用户输入国别 + 企业关键词,系统返回候选企业列表,用户选定后进入详情页查看:

- 四维评分(基础工商 15 / 资质认证 25 / 财务健康 30 / 司法舆情 30)
- A/B/C/D 等级
- 四维雷达图
- 基本信息卡片
- AI 综合评价(缓存)
- AI 对话追问(实时流式)

### 1.2 数据来源边界(关键)

**第一阶段所有外部数据全部 mock,通过 seed 写入数据库**。

- 4 家 demo 企业的所有数据(工商基础、财务、司法舆情、证书)由 seed 脚本生成
- 评分计算逻辑真实运行(读 mock 数据 → 走规则引擎 → 写真实快照)
- AI 评价真实调用千问 API(基于 mock 评分数据生成)
- 候选列表搜索仅查本地 `credit_company` 表,不调外部 API

**这意味着第一阶段成果是"产品形态的素描":数据假,但产品骨架、评分引擎、AI 能力都是真的。**真实数据源接入由 TODO 清单管理,待 PM 与数据供应商商务对接完成后启动。

### 1.3 触发模式

- 第一阶段:**人工触发 + seed 预置**。提供 `POST /credit/recompute` 接口由 OPERATOR 手动触发重算。无后台调度器。
- 长期(T+1):TODO 标记,引入调度后启用。

## 三、抽象层设计

按"可扩展、高内聚、低耦合"原则,本期建立 4 个抽象层,代码组织如下:

  ├── data_source/
  │     ├── base.py              # DataSource 抽象基类

### 3.1 DataSource 抽象层

**职责**:封装"如何获取某公司某类数据"。第一阶段实现是 `MockDataSource`,直接读已 seed 到 `credit_company_*_data` 表的数据。

**接口**:

**长期(T-2)**:新增 `QichachaDataSource` / `OpenCorporatesDataSource` 等,通过 `registry.py` 按国别路由。

### 3.2 ScoringEngine 评分引擎

**职责**:输入一个公司 ID,输出一份完整 `score_snapshot` + `score_detail`,可选触发 `score_audit_log`。

**核心流程**:

```
1. 加载 active 规则集:
     · score_dimension / score_subitem / score_rule(子项级)
     · score_dimension_override(维度级)
2. 通过 DataSource 获取该公司四类数据 + 证书清单
3. 子项级自然评分:
     对每个 active subitem:
       a. 取该 subitem 下所有 active rule,按 priority 升序
       b. 依次调用 SUBITEM_EVALUATORS[rule.evaluator_key](data)
       c. 首条返回 true 的 rule,该 subitem 得 rule.score,记录 hit_rule
       d. 全部未命中,走 subitem.default_score
4. 维度自然分:四个维度的 subitem 自然得分各自求和 → dimension_N_natural_score
5. 维度级 override post-process:
     对每个维度:
       a. 取该 dimension 下所有 active dimension_override,按 priority 升序
       b. 依次调用 DIMENSION_OVERRIDE_EVALUATORS[override.evaluator_key](data)
       c. 首条命中:
            · dimension_N_score = override.override_score(覆盖自然分)
            · dimension_overrides 数组追加一条记录
       d. 无命中:dimension_N_score = dimension_N_natural_score
6. 总分 = sum(四个维度 final score);grade 阈值映射(A≥80/B≥60/C≥40/D<40)
7. 写入 score_snapshot,含 natural_score + final_score + dimension_overrides
8. 写入 12 条 score_detail(hit_rule 与 score 均为自然评分结果,不受 override 影响)
9. 对比前一快照,如分数有变化,写入 score_audit_log + 平台 audit_logs
10. 异步调 AISummaryGenerator 生成 ai_summary 回写
```

**事务边界**:步骤 7-9 在同一数据库事务内提交,保证"快照写入 + is_current 切换 + 明细写入 + 审计写入"原子性。步骤 10 异步执行,失败不回滚评分结果。

**与 score_detail 的设计契约**:被 override 的维度,detail 表 3 条记录保留**自然命中规则**和**自然得分**,与 snapshot 的 `dimension_N_score`(override 后)可能不一致。这是数据建模层面的设计,供未来如有需要追溯自然分时使用,不预设 UI 呈现。

ScoringEngine 不直接耦合任何外部服务,可独立单测(给定 mock 数据 → 检查输出快照)。

### 3.3 Evaluators 规则求值函数

集中在 `app/services/credit/evaluators.py`,所有函数签名统一为 `fn(data: dict) -> bool`,`data` 是 ScoringEngine 准备好的统一上下文,包含该公司四类数据 + 证书清单。

文件内分为两组互不混用的注册表:

```python
# === 子项级 evaluators(对应 score_rule.evaluator_key)===

SUBITEM_EVALUATORS: dict[str, Callable] = {
    "basic_reg_info_full": basic_reg_info_full,
    "basic_reg_info_miss_one": basic_reg_info_miss_one,
    # ... 共约 35 个,对应 PRD §4.3 各子项的所有自然规则档位
}

# === 维度级 override evaluators(对应 score_dimension_override.evaluator_key)===

DIMENSION_OVERRIDE_EVALUATORS: dict[str, Callable] = {
    "dim2_cert_forged_or_expired": dim2_cert_forged_or_expired,
    "dim3_unknown": dim3_unknown,
    "dim4_unresolved_defaulter": dim4_unresolved_defaulter,
}
```

ScoringEngine 在子项级求值步骤只查 SUBITEM_EVALUATORS,在维度级 post-process 步骤只查 DIMENSION_OVERRIDE_EVALUATORS。两组不交叉,避免 evaluator_key 在不同语境下被误用。

### 3.4 LLMService 抽象层

见 ADR-0003。MVP 实现 `QwenChatService`,提供两个方法:

- `generate(prompt: str) -> str`:同步调用,用于 AI 综合评价
- `stream_chat(messages: list[dict]) -> AsyncIterator[str]`:流式,用于对话追问

### 3.5 AISummaryGenerator

**职责**:基于 score_snapshot + score_detail 生成 ai_summary。

**Prompt 模板**(代码内常量):

```
你是海外工程供应链领域的企业风控分析师。
基于以下企业的结构化评分数据,生成约 200-400 字的综合评价。
要点:
  ① 综合得分与等级解读
  ② 优势(高分维度)
  ③ 风险提示(低分/数据缺失维度)
  ④ 合作建议
要求:语气专业客观,不使用绝对化措辞。若某子项数据状态为 missing,如实说明而不编造。

【企业数据】
{snapshot_json}
```

调用失败时不阻塞评分写入,`ai_summary` 字段留 null,前端展示"AI 评价正在生成"。

## 六、RBAC 与权限

权限设计严格遵循 `RBAC设计规范_v1_0.md` §8 数据权限设计原则:**权限点不带 scope 后缀,数据范围在服务层通过查表函数决策**。

### 6.1 权限点定义

挂到现有 `app/rbac/constants.py`,模块归类到业务-档案(`BIZ_ARCHIVE`):

| 权限点 code | 名称 | 说明 |
|---|---|---|
| `credit:read` | 查看信用评估 | 列表 + 详情 + AI 对话相关 |
| `credit:write` | 维护企业档案 | 本期不暴露 API,留位 |
| `credit:recompute` | 触发评分重算 | 单家重算 + 全量重算 |

### 6.2 角色 × 权限点 × scope 映射

| 角色 | credit:read | credit:write | credit:recompute |
|---|---|---|---|
| BUYER(央企采购员) | ✅ scope=ALL | ❌ | ❌ |
| OPERATOR(平台运营) | ✅ scope=ALL | ✅ scope=ALL | ✅ scope=ALL |
| ADMIN(系统管理) | ❌ | ❌ | ❌ |
| SUPPLIER(平台入驻供应商) | ✅ scope=OWN | ❌ | ❌ |

ADMIN 作为系统管理角色,按 RBAC 规范 §4.3 + §8.6 职责分离原则,**不持有任何业务数据相关权限点**,只在系统数据(用户/角色/权限/配置/日志)上有权限。

### 6.3 scope=OWN 的归属字段

按 RBAC 规范 §8.4 "组织级归属":

- 归属字段:`credit_company.linked_supplier_org_id`
- 取当前 user 所属的 supplier_org_id(通过 Membership 表查),作为过滤条件
- SQL 形式:`WHERE credit_company.linked_supplier_org_id = :user_supplier_org_id`

### 6.4 scope 决策与接口实现

按规范 §8.5.3,在统一的 scope 决策函数中加入信用评估模块的映射条目。接口实现严格遵循"两层校验"模式:

```
require_permission('credit:read')         # 第一层:动作许可
scope = get_scope(user, 'credit:read')    # 第二层:数据边界
按 scope 组装 WHERE 条件
```

### 6.5 不命中数据的接口行为

按规范 §8.3,scope=OWN 过滤后未命中数据:

| 接口 | 返回 |
|---|---|
| 列表类(`/credit/companies/search`) | 空数组 |
| 详情类(`/credit/companies/{id}`) | 404(不暴露存在性) |
| AI 对话类(`/credit/ai/conversations/*`) | 404 |
| 重算类(权限点已拦截) | 403 |

### 6.6 第一阶段 SUPPLIER 数据范围说明

第一阶段 `credit_company` 表中无 SUPPLIER 镜像数据(`linked_supplier_org_id` 全为 NULL,4 家 demo 企业均为平台外目标公司)。SUPPLIER 登录后:

- scope=OWN 过滤后,所有接口返回空/404
- 前端 `/credit` 页面正常展示,但数据列表为空
- SUPPLIER 评分功能完整接入(注册时建镜像 + 触发首次评分 + 数据范围控制)的产品决策尚未确定,见 §10.6

前端 `/credit` 路由从当前的 PublicLayout 改为登录可见,跟权限点对齐。

### 10.4 Seed 数据更新策略

本期 seed 采用**"无则插入,有则不动"**幂等策略:启动时检查每条记录的 code 是否已存在,存在则跳过,不存在则插入。

**该策略下,规则元数据变更(score、description、priority 等)不会自动同步到已存在的数据库记录**。规则变更后,需要其中之一:

- 手动执行 SQL UPDATE 同步已 seed 记录
- 重置数据库后重新启动(开发环境可接受,生产不可)
- 写 alembic data migration 显式更新(标准做法,但本期未启用)

第一阶段所有数据为 mock,无运营介入,无生产环境部署,该限制可接受。规则变更频率上升后,需启用正式的规则更新机制,见 §九 T-3。
