> **评审日期**:2026-05-19
> **评审基线**:`main @ da20f0f`(截至 PR #11 / #12 合并完成,T1–T6 全部入主)
> **评审视角**:高级架构师
> **范围**:认证 / RBAC / 审计底座(后端 + 前端),不含尚未实现的业务模块
> **目的**:盘点架构层面隐患,按 P0 → P3 排优先级,给"现在动"和"以后再说"划清边界
> **两条评审视角**:
> - **§2-§9**:**工程运维角度** —— 部署 / 监控 / 扩容(P0-P3)

## 0. 总体判定

架构骨架是**对**的(分层清晰、RBAC v3 配置即代码、auth/audit 底座做得扎实),但存在 **4 个 P0 级架构隐患**,会让这套系统在第一次真正上线时出事。其余是 tech debt 范畴。

MVP 阶段问题不大,但项目定位是"央企海外平台",标准应该对齐**正式系统**而不是"先跑起来再说"。

---

## 1. 做得对的地方

非穷举,只点名:

- **分层清晰**:`api → service → db`,业务逻辑没渗到路由层
- **RBAC v3 双层 scope** 设计精彩:`permissions_config.py` + `scope_config.py` + 启动同步 = **配置即代码**,后续加权限零迁移
- **httpOnly refresh cookie + 内存 access token** 是 2025 年安全最佳实践
- **Alembic 迁移可逆**,`downgrade()` 都实写、跑过往返
- **Trace ID 中间件 + audit_logs 全链路串联**(只差结构化日志这一步)
- `ValidationFailedError → 400` 与 pydantic 422 的边界明确
- 审计写失败不阻断主流程的可用性选择(虽然下面要批评这点的合规面)
- 测试 fixture `drop_all/create_all` 隔离干净
- 注册不自动登录 —— 行业规范,坚持得好
- ADMIN 严格不触业务数据 —— Q25 决策正确,符合等保职责分离
- 双 Organization 表(BuyerOrg / SupplierOrg) —— 业界主流(Salesforce / SAP 也这样)
- 4 角色 × 35 权限点的范围 —— MVP 该停的地方停住了
- 审计 GET 不写 —— 噪音控制好

## 2. P0 · 上线前必须解决

### P0-1 · `alembic upgrade head` 写在应用启动里

**现状**:`docker-compose.yml` 中 backend 容器的 `CMD` 是 `alembic upgrade head && uvicorn ...`。
**风险**:多实例部署时,所有 worker 同时改 schema,锁表死锁是迟早。

**架构错位**:迁移是 **CI/CD 的一步**,不是 app 启动的一部分。

**修法**:

- CI 跑 `alembic upgrade head`,通过才部署
- app 启动只做 **head 校验**,不一致直接退出:

```python
# main.py lifespan 开头
from alembic.runtime.migration import MigrationContext
from alembic.script import ScriptDirectory

cfg = AlembicConfig("alembic.ini")
script = ScriptDirectory.from_config(cfg)
async with engine.connect() as conn:
    ctx = MigrationContext.configure(conn.sync_connection)
    current = ctx.get_current_revision()
head = script.get_current_head()
if current != head:
    raise RuntimeError(f"DB schema {current} != app expects {head}")
```

### P0-2 · `ENABLE_DEBUG_API: bool = True` 默认开

**现状**:`app/core/config.py:46` 默认值 `True`。`/_debug/scope` `/_debug/matrix` 在生产泄露 → 内部 RBAC 矩阵全曝。
**风险**:部署模板很容易漏改这个;一次性失误就把整个权限设计暴露给攻击者做映射。

**修法**:默认 `False`,本地 `.env` 显式 `=true`。已有先例(`SEED_DEMO_ACCOUNTS` 就是默认关、显式开),按同样思路统一。

---

### P0-3 · 单机内存 rate limiter

**现状**:`services/rate_limit.py` 是模块级 dict。
**风险**:两个 worker 起来限流就被绕过。`CLAUDE.md` 写了"MVP 单机足够"但项目已经在考虑央企海外业务量,这就是埋着的雷。

**修法**:不要急着上 Redis(项目禁了),但要**清晰地把限流抽成 interface**,实现层是 in-process dict。到了要上 Redis 那天只换实现,业务代码零改动。

```python
class RateLimiter(Protocol):
    def is_locked(self, key: str, ip: str) -> bool: ...
    def record_failure(self, key: str, ip: str) -> bool: ...
    def reset(self, key: str, ip: str) -> None: ...

class InMemoryRateLimiter(RateLimiter): ...   # 当前实现
# 未来 class RedisRateLimiter(RateLimiter): ...
```

---

### P0-4 · 启动时 RBAC 同步在多实例下竞态

**现状**:`rbac/sync.py` 启动跑 `INSERT/DELETE` 到 `permissions` / `role_permissions`。
**风险**:两个 pod 同时启 → 同时跑 sync → 唯一约束冲突的处理逻辑未经多实例验证。

**修法**(二选一):

- **方案 A**(推荐):同 P0-1 一起进 CI 阶段,CI 跑 sync,失败 fail deploy
- **方案 B**:加 PG **advisory lock**

```python
# sync.py 入口
await conn.execute(text("SELECT pg_advisory_lock(:k)"), {"k": hash("rbac_sync") & 0x7fffffff})
try:
    await do_sync(...)
finally:
    await conn.execute(text("SELECT pg_advisory_unlock(:k)"), {"k": ...})
```

## 3. P1 · 业务量起来前必须想清楚

### P1-1 · `scope` 配置完整但 service 层没人用 ← **最危险的一条**

**现状**:`scope_config.py` 定义了 BUYER 看 `OWN_ORG`、SUPPLIER 看 `OWN`,**但整个仓库里没一行 service 代码真的拿 scope 拼 WHERE**(因为业务表还没上)。

**风险**:业务模块开发时 dev 忘记走 scope filter,**数据越权读直接发生**。前端守卫 + 后端 Guard 都只校验"能不能做",**不校验"对谁做"**。

**修法**:在 business service 上线**之前**,写一个强制 helper:

```python
def apply_scope[T](
    q: Select[T],
    *,
    resource: str,
    user: CurrentUser,
    owner_col: ColumnElement[int],
) -> Select[T]:
    scope = scope_of(user.roles, resource)
    if scope == "OWN_ORG": return q.where(owner_col == user.buyer_org_id)
    if scope == "OWN":     return q.where(owner_col == user.id)
    if scope == "ALL":     return q
    if scope == "NONE":    raise PermissionDeniedError(...)
```

所有业务 service 必须强制走它。**配 lint 规则**:`select(<BusinessModel>)` 不经过 `apply_scope` 就 fail。

> 这是当前 RBAC 设计最大的资产。现在它只活在调试接口和文档里 —— 业务模块上线那天必须**先 ship `apply_scope` helper + lint 规则**,而不是急着写 RFQ。

---

### P1-2 · JWT 静态密钥 + HS256

**现状**:单一对称密钥,泄一次全员重发。
**风险**:央企客户做安全审计这一项基本过不了。

**修法**:换 **RS256** + 密钥轮换。`/auth/jwks` 暴露 public key。即使 MVP 不做轮换,先把基础设施改 RS256,后续无痛。

---

### P1-3 · `audit_logs` 无限增长

**现状**:每天上千条 trivial(LOGIN_FAILED + LOGIN_LOCKED 包括在内),一年后表能 GB 级,索引开始慢。

**修法**:

- 按月分区(PG declarative partitioning,创建迁移即可)
- 加 retention 策略:**90 天**前归档到冷表,例如 `audit_logs_archive`

---

### P1-4 · `/healthz` 不探 DB

**现状**:

```python
@router.get("/healthz")
async def health(): return success({"status": "ok"})
```

**风险**:LB 看这个全绿,但 PG 可能已经挂,流量持续打到坏实例。

**修法**:分两个端点

```python
@router.get("/healthz")  # 进程存活
async def health(): return success({"status": "ok"})

@router.get("/readyz")  # 真就绪
async def ready(db: AsyncSession = Depends(get_db)):
    await db.execute(text("SELECT 1"))
    return success({"status": "ok"})
```

LB readiness 探针指向 `/readyz`,liveness 指向 `/healthz`。

---

### P1-5 · 审计写失败"静默吃掉"

**现状**:`write_audit` 失败时,主流程继续。
**风险**:等保 2.0 三级"安全审计"项可能被扣分;合规审计时丢事件无回溯。

**修法**:写不进 DB 时,**落本地文件** + 日志双写作为兜底:

```python
try:
    db.add(AuditLog(**payload))
    await db.commit()
except Exception:
    logger.error("AUDIT_DB_FAIL", extra={"audit_payload": payload})
    # 兜底写本地 jsonl 文件,运维至少能反找回来
    with open("/var/log/app/audit-fallback.jsonl", "a") as f:
        f.write(json.dumps(payload) + "\n")
```

## 10. 架构逻辑层走歪问题(独立维度)

> §2-§9 看的是"**工程化能不能扛**"。本节看的是"**概念上有没有自洽**"。
>
> 两种问题不可互相替换:运维问题可以靠基础设施补,逻辑走歪只能靠重构。

### 10.1 真走歪的 6 条

#### W1 · ADMIN 通过 `system:audit` 间接看业务数据 → 与 Q25 冲突 `[合规级]`

`ADMIN` 角色定义是**严格不触业务数据**(Q25 决策),但同时给了 `system:audit` 权限,而审计日志里**全是业务操作**(谁创建供应商、谁停用账号、谁审批 RFQ)。

ADMIN 打开 `/admin/audit-logs` 就能看到全平台业务行为流水。**Q25 名存实亡**。

等保 2.0 三级原则:**审计员**应该是独立角色,与"系统管理员"分离,互相监督。现在这两个角色合并了,系统管理员可以查看(虽然权限点上没给出删除审计的能力)自己的操作记录,没有制衡。

**应有**:拆出第 5 个角色 `AUDITOR`,持有 `system:audit`,**ADMIN 失去这个权限点**。或者把 `system:audit` 拆成 `audit:read`(给 AUDITOR)和 `audit:export`(给监管对接用)。

---

#### W2 · scope 模型只到 `(role, resource)` 二元,缺 `action` 维度 `[模型表达力级]`

**现状**:`scope_config.py` 是 `Map<(Role, Resource), Scope>`。

**问题**:一个 BUYER 对 `project` 资源,**所有动作共享同一个 scope**。但真实业务里:

- BUYER **read** project → `OWN_ORG`(看全公司项目)
- BUYER **write** project → `OWN`(只能改自己创建的)
- BUYER **delete** project → 也许 `NONE`(只能停,不能删)

二元映射表达不了。**第一次业务需求来了就得改模型**。

**应有**:`Map<(Role, Resource, Action), Scope>`,或者更紧凑:`Map<(Role, PermissionCode), Scope>`(因为 PermissionCode 已经含 resource:action)。

不是大改 —— 配置文件扩一列,scope 同步逻辑改成读三元就行。**但现在不动,业务模块上线第二个礼拜就要返工**。

---

#### W3 · ADMIN 能停业务账号(BUYER / SUPPLIER)→ 职责越界 `[职责越界级]`

**现状**:`POST /admin/users/{id}/disable` 守卫 `user:manage`,可以停**任何账号**,包括 BUYER / SUPPLIER。

**逻辑问题**:ADMIN 不应触业务数据(Q25)。但停 BUYER 账号本质上是**业务决定**("这个采购员离职了" / "这个供应商有合规问题")。ADMIN 看不到业务数据,**他凭什么决定要停谁**?盲停?

应该是 OPERATOR 决定停业务账号(基于业务原因),ADMIN 只动 ADMIN / OPERATOR 账号(系统层操作)。

**应有**:`user:manage` 拆 → `user:manage:internal`(ADMIN / OPERATOR 账号)与 `user:manage:business`(BUYER / SUPPLIER 账号)。前者给 ADMIN,后者给 OPERATOR。

---

#### W4 · `User → Role` 与 `User → BuyerMember/SupplierMember` 数据完整性约束跨表无强制 `[数据可靠级]`

**问题**:语义上一个挂 BUYER 角色的用户**必须**有 BuyerMember 记录;挂 SUPPLIER 必须有 SupplierMember。但 schema 层**没有任何强制**。

后果:运维直接 SQL 造一个 user 挂 BUYER 角色但漏建 BuyerMember,系统就**分裂态**:

- `/auth/me` 返回 `roles=["BUYER"]`, `organization=null`
- RBAC 给 BUYER 权限通过
- 但 service 层做 `apply_scope(... owner_col = buyer_org_id)` 时 `user.buyer_org_id` 是 null → SQL `WHERE buyer_org_id = NULL` 永远不匹配 → **业务接口全失败但不报"无权限"**

**应有**:

- 应用层:`/auth/me` 检测到这种分裂态时抛 500 + 写审计 `ORPHAN_USER`,而不是返回半空数据
- 注册 service 用单事务包裹 User + Member + UserRole 创建,失败回滚(目前已经是单事务,OK)
- 长期:在 `audit_logs` 之外定期跑一致性 check job

这条不能用 PG 约束强制(跨多表),只能应用层防御。

---

#### W5 · `auth_service.py` 职责过宽 → "认证"和"账号管理"混在一起 `[可维护级]`

**现状**:`auth_service.py` 350 行,塞了:

- `login` / `logout` ← 真·认证
- `register_buyer` / `register_supplier` ← 用户创建 + 组织挂载 + 角色分配(**这是账号管理 + 业务**)
- `change_password` ← 凭证管理(更适合 me_service)
- `_find_user_by_identifier` ← 工具,挪 utils

`me_service.py` 已经收了 `change_email` / `change_username` / `change_phone`。**`change_password` 不在那里,逻辑分裂**。

`register_*` 实际上是**业务流程**(创建组织、生成审计、赋角色),不是"认证服务"。

**应有**:

```
auth_service          → login / logout / token refresh / _find_user_by_identifier
me_service            → change_password / change_email / change_username / change_phone / update_profile
registration_service  → register_buyer / register_supplier(含组织挂载)
```

这条改了不影响功能,但代码可读性 + 测试边界都会清晰得多。

---

#### W6 · 公开池(`/mall` `/suppliers`)与 RBAC scope 模型冲突 `[概念缺失级]`

**现状**:`/mall`、`/suppliers`、`/countries` 是公开页(PUBLIC_NAV),不登录可访。

**矛盾点**:

- 后端 RBAC 模型里,SUPPLIER 资源对未登录用户 scope = `NONE`
- 但 `/suppliers` 公开页明明要展示"已上架供应商目录"
- 前端 PermissionPlaceholderPage 在公开页上还在跑 scope 检测,得到 `NONE` → 显示"维度 3 红色:无访问权"

更根本:**scope 模型里没有"公开池"概念**。每个资源只有 `OWN` / `OWN_ORG` / `ALL` / `NONE` 四档,缺 `PUBLIC` 那档。

**应有**:scope 加一档 `PUBLIC`(=未登录也能读,但只读已审核/已上架的子集),或者**业务表加 `is_published` 字段**,公开池查询走 `WHERE is_published=true` 不走 scope filter。

这个走歪程度比前面 5 条小,但定义业务模块时绕不开。

---

### 10.2 边缘问题(设计权衡 vs 错误)

#### E1 · `BuyerMember` / `SupplierMember` 双表 vs 统一 `memberships`

双表结构几乎完全一样(`user_id / org_id / is_owner / created_at`),只是引用不同 org。

- 双表的好处:类型安全,FK 清晰
- 双表的坏处:"用户在哪些组织"查询要 UNION;加新组织类型(集团采购联合体?)又要加一张表

不算错,但你已经为业务属性差异(企业资料 / 资质 / 审核流程)分了 `BuyerOrganization` / `SupplierOrganization` 两张表 —— **member 层其实可以合并**,因为 member 关系本身没有业务属性差异。

#### E2 · `BuyerOrganization.code` 字段是死代码

`code` 字段最初是 seed 用的(`CSCEC3B`),T2 之后注册走 `unified_social_credit_code` 识别。`code` 没人用了。

要么删字段,要么明确文档"内部展示别名"。**保留不明确用途的字段会让后人不敢删**。

#### E3 · `must_change_password` 强制由前端还是后端拦?

设计文档说"业务接口被拦"。但实际拦在哪没验证。如果只前端 RouteGuard 拦,**API 直调可绕过**。

应该是后端中间件:`must_change_password=true` 时,除白名单(`/auth/change-password` / `/auth/me` / `/auth/logout`)外全 403。需要 grep 验证现状。

#### E4 · supplier 重复入驻新文案"如需加入该企业,请联系企业管理员"

"加入企业"功能**当前不存在**(归第三阶段)。文案承诺了 UX 但系统能力不到位 → 用户期望落差。

短期 OK,长期要么把"邀请员工"做出来,要么文案改更模糊。
