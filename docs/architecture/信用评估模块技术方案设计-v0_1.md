## 一、定位与边界

### 1.1 定位

第一阶段信用评估模块是"海外工程领域专业版企查查"。用户输入国别 + 企业关键词,系统返回候选企业列表,用户选定后进入详情页查看:

- 四维评分(基础工商 15 / 资质认证 25 / 财务健康 30 / 司法舆情 30)
- A/B/C/D 等级
- 四维雷达图
- 基本信息卡片
- AI 综合评价(缓存)
- AI 对话追问(实时流式)

### 1.2 数据来源边界(关键)

**第一阶段所有外部数据全部 mock,通过 seed 写入数据库**。

- 4 家 demo 企业的所有数据(工商基础、财务、司法舆情、证书)由 seed 脚本生成
- 评分计算逻辑真实运行(读 mock 数据 → 走规则引擎 → 写真实快照)
- AI 评价真实调用千问 API(基于 mock 评分数据生成)
- 候选列表搜索仅查本地 `credit_company` 表,不调外部 API

**这意味着第一阶段成果是"产品形态的素描":数据假,但产品骨架、评分引擎、AI 能力都是真的。**真实数据源接入由 TODO 清单管理,待 PM 与数据供应商商务对接完成后启动。

### 1.3 触发模式

- 第一阶段:**人工触发 + seed 预置**。提供 `POST /credit/recompute` 接口由 OPERATOR/ADMIN 手动触发重算。无后台调度器。
- 长期(T+1):TODO 标记,引入调度后启用。

## 三、抽象层设计

按"可扩展、高内聚、低耦合"原则,本期建立 4 个抽象层,代码组织如下:

```
app/services/credit/
  ├── data_source/
  │     ├── base.py              # DataSource 抽象基类
  │     ├── mock_data_source.py  # 第一阶段实现:读 seed 数据
  │     └── registry.py          # 按国别路由数据源(预留)
  ├── evaluators.py              # 规则求值函数集中地
  ├── scoring_engine.py          # 评分引擎(读规则 + 调求值器 + 写快照)
  └── ai_summary_generator.py    # AI 综合评价生成器(调 LLMService)

app/services/llm/
  ├── base.py                    # LLMService 抽象基类
  └── qwen_chat_service.py       # 千问实现
```

### 3.1 DataSource 抽象层

**职责**:封装"如何获取某公司某类数据"。第一阶段实现是 `MockDataSource`,直接读已 seed 到 `credit_company_*_data` 表的数据。

**接口**:

```python
class DataSource(ABC):
    @abstractmethod
    async def fetch_basic_data(self, company_id: int) -> BasicData: ...
    @abstractmethod
    async def fetch_finance_data(self, company_id: int) -> FinanceData: ...
    @abstractmethod
    async def fetch_legal_data(self, company_id: int) -> LegalData: ...
    @abstractmethod
    async def fetch_certifications(self, company_id: int) -> list[Certification]: ...
```

**第一阶段**:`MockDataSource` 实现 = 直接 SELECT 最新一条 `credit_company_*_data`。

**长期(T-2)**:新增 `QichachaDataSource` / `OpenCorporatesDataSource` 等,通过 `registry.py` 按国别路由。

### 3.2 ScoringEngine 评分引擎

**职责**:输入一个公司 ID,输出一份完整 `score_snapshot` + `score_detail`,可选触发 `score_audit_log`。

**核心流程**:

```
1. 加载 active 规则集(dimension/subitem/rule 全表 cache,版本一致)
2. 通过 DataSource 获取该公司四类数据
3. 对每个 subitem:
   a. 取该 subitem 下所有 active rule,按 priority 升序
   b. 依次调用 evaluator(从 EVALUATORS 字典查 evaluator_key)
   c. 首条返回 true 的 rule,该 subitem 得 rule.score
   d. 全部未命中,走 subitem.default_score
4. 维度得分 = 该维度下所有 subitem 得分之和
5. 总分 = 四维度之和;grade 按阈值映射(A≥80/B≥60/C≥40/D<40)
6. 写入 score_snapshot(is_current=true,旧的置 false)
7. 写入 12 条 score_detail
8. 对比前一快照,如分数有变化,写入 score_audit_log + 平台 audit_logs
9. 异步调 AISummaryGenerator 生成 ai_summary 回写
```

**事务边界**:步骤 6-8 在同一数据库事务内提交,保证"快照写入 + is_current 切换 + 明细写入 + 审计写入"原子性。步骤 9 异步执行,失败不回滚评分结果。

ScoringEngine 不直接耦合任何外部服务,可独立单测(给定 mock 数据 → 检查输出快照)。

**维度级 override 的实现**:PRD §4.3 三个维度级强制规则——
- 维度2 "关键证书伪造/过期 → 维度强制清零"
- 维度3 "整维度数据缺失 → 该维度 40% 满分(12 分)"
- 维度4 "失信被执行未结案 → 维度直接判 0"(一票否决)

均通过**给该维度内的每个 subitem 各挂一条 priority=0 的 score_rule** 实现,共 9 条 override rule(3 维度 × 3 子项)。priority=0 比常规规则的 10/20/30/40 优先级高,在步骤 3 的求值循环里**首先**被检查,命中后该 subitem 取 override rule 的 score(0 或 4),自然命中规则不再触发。结果是:被 override 时,该维度的 3 个 subitem 评分一致(都被压成 0 或 4),score_detail 的 hit_rule_description 字面上就是 override 描述。这种实现保留"subitem 级 rule 命中"的统一语义,不需要在 engine 里加维度级 post-process 钩子,但**代价**是 score_detail 丢失"自然命中规则"信息(若未来需要展示"被 override 之前原本会得多少分",需重构为 engine 内分两步:先 natural evaluate → 再维度级 override post-process)。

ScoreSnapshot 的 `basic_data_id` / `finance_data_id` / `legal_data_id` 引用本次评分使用的数据快照行。**当对应类型的数据缺失(`data_source == "missing"`)时该字段为 NULL**;UI 详情接口据此判断是否展示对应卡片。

### 3.3 Evaluators 规则求值函数

集中在 `app/services/credit/evaluators.py`,所有函数签名统一:

```python

    "basic_reg_info_full": basic_reg_info_full,
    "basic_reg_info_miss_one": basic_reg_info_miss_one,

```

`data` 参数是 ScoringEngine 准备好的统一上下文,包含该公司四类数据。

### 3.4 LLMService 抽象层

见 ADR-0003。MVP 实现 `QwenChatService`,提供两个方法:

- `generate(prompt: str) -> str`:同步调用,用于 AI 综合评价
- `stream_chat(messages: list[dict]) -> AsyncIterator[str]`:流式,用于对话追问

**错误信息脱敏策略**:LLM 上游调用失败(配置缺失 / 鉴权失败 / 限流 / 超时 / 服务端错误)统一抛 `LLMUnavailableError`。该异常的 message 字段**仅用于服务端日志**,绝不允许把厂商名 / 模型名 / 环境变量名 / 原始上游 error string 透传到客户端。所有面向前端的错误出口(详情接口、SSE 流式接口的 `event: error` 数据帧、AI 综合评价空值降级)统一用通用文案,如"AI 服务暂时不可用,请稍后再试"或"AI 评价暂时不可用"。具体原因走 `logger.warning` / `logger.exception` 落服务端日志(含 trace_id)。

### 3.5 AISummaryGenerator

**职责**:基于 score_snapshot + score_detail 生成 ai_summary。

**Prompt 模板**(代码内常量):

```
你是海外工程供应链领域的企业风控分析师。
基于以下企业的结构化评分数据,生成约 200-400 字的综合评价。
要点:
  ① 综合得分与等级解读
  ② 优势(高分维度)
  ③ 风险提示(低分/数据缺失维度)
  ④ 合作建议
要求:语气专业客观,不使用绝对化措辞。若某子项数据状态为 missing,如实说明而不编造。

【企业数据】
{snapshot_json}
```

调用失败时不阻塞评分写入,`ai_summary` 字段留 null,前端展示"AI 评价正在生成"。

## 五、关键时序

### 5.1 用户查看评分

```
[用户搜索] GET /credit/companies/search
  → 本地 credit_company 模糊查 → 返回候选(含分/等级)
  → TODO(T-2): 本地无结果时调外部 API 兜底

[用户选定] GET /credit/companies/{id}
  → 写 credit_search_history
  → 一次查询返回:
     · credit_company 标识字段
     · credit_company_basic_data 最新一条(用于"基本信息卡"展示)
     · 最新 is_current=true 的 score_snapshot
     · 该 snapshot 关联的 12 条 score_detail
     · credit_company_certification 全部记录
  → 返回完整数据(含已缓存的 ai_summary)

[用户追问] POST /credit/ai/conversations/{id}/messages
  → 写 user 消息 → 拼装上下文(企业数据 + 历史消息)→ 调 QwenChatService.stream_chat
  → SSE 流式返回 → 流完写 assistant 消息
```

### 5.2 评分计算(人工触发版)

```
POST /credit/companies/{id}/recompute
  → ScoringEngine.compute(company_id)
      → DataSource 取四类数据
      → 加载规则集
      → 子项级求值
      → 写 score_snapshot(新)+ 切换 is_current
      → 写 12 条 score_detail
      → 对比前快照,如变化:写 score_audit_log + 平台 audit_logs
  → 同步调 AISummaryGenerator → 写 ai_summary
  → 返回新快照
```

### 5.3 长期完整流程(TODO,仅作架构对齐)

```
[T+1 跑批]
  遍历 credit_company → 调 DataSource 抓最新数据(写入快照表)
  → ScoringEngine 重算 → AISummaryGenerator → 完成

[用户搜到库外公司(T-2)]
  本地无结果 → 调 ExternalDataSource → 候选列表
```
