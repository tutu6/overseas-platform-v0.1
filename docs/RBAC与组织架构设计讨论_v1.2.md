## 一、业务背景

| 背景 | 对设计的影响 |
|---|---|
| 主要业主是中建三局(初期单一大客户) | 多租户压力小,但需为"未来加入其他央企"留口子 |
| 功能 2-3 个月内不会大变动 | 权限关系可"先固化",不急于做配置后台 |
| 供应商上限千级、采购方四位数 | 量级小,**不需要权限缓存、不需要分库分表** |
| 并发不高 | 权限校验直接查库,无性能压力 |

**核心结论**:这个量级下,**过度工程化是最大的浪费**。RBAC 要做规范,但不做花哨。

---

## 二、整体定位

### 复杂度定档:标准 RBAC0,认真做

不偷懒(不用 if-else 伪 RBAC),也不过度(不做角色继承 / SOD / ABAC / 字段权限)。

### 砍掉的部分

| 不做 | 原因 |
|---|---|
| ❌ 角色继承(RBAC1) | MVP 角色之间无继承关系 |
| ❌ 职责分离 SOD 规则引擎 | 用"职责自然分离"(operator/admin 拆分)替代 |
| ❌ ABAC(基于属性的权限) | 业务复杂度远未到 |
| ❌ 字段级权限 | L3 档能力,MVP 不必 |
| ❌ 权限缓存层 | 量级不足 |
| ❌ 权限管理后台 UI | 通过 seed/配置足够,V1 再做 |
| ❌ 数据权限策略引擎 | 用归属字段硬过滤即可 |

## 三、两个核心维度:组织实体 vs 系统角色

⚠️ **这是 RBAC 设计的第一原则,前置阅读**。不要把"组织"和"角色"混在一起,它们是**两个独立维度**:

### 3.1 维度图

```
                    ┌──────────────────────┐
                    │   User (登录身份)     │
                    └──────────┬───────────┘
                               │
              ┌────────────────┴────────────────┐
              ↓                                 ↓
   【维度 A:组织归属】                  【维度 B:系统角色】
              │                                 │
   ┌──────────┴───────────┐         ┌──────────┴──────────┐
   ↓                      ↓         ↓                     ↓
 BuyerOrg          SupplierOrg     Role 表(4 个固定)
 (中建三局)         (N 家供货商)
   │                      │          PURCHASER  (挂 BuyerOrg)
   ↓                      ↓          SUPPLIER   (挂 SupplierOrg)
 BuyerMember     SupplierMember     OPERATOR   (不挂组织)
 (用户在组织内)    (用户在组织内)     ADMIN      (不挂组织)
```

### 3.2 两个维度的本质

| 维度 | 数量 | 回答的问题 |
|---|---|---|
| 组织实体 | 2 个 | 用户**属于哪家公司**? |
| 系统角色 | 4 个 | 用户**能做什么**? |

两者独立配置,通过 User 关联起来。

### 3.3 类比

> 一张身份证(User)上面同时有两个独立信息:
> - **籍贯**(组织实体):你是哪里的人
> - **职业**(系统角色):你做什么工作
>
> 这两个信息没有强绑定关系——同样是"北京籍",有人是医生,有人是教师。

## 四、组织模型设计

### 4.1 决策:双组织实体设计 ⭐

| 实体 | 说明 |
|---|---|
| `BuyerOrganization` | 采购方组织(央企业主),MVP 阶段只有"中建三局"1 条数据 |
| `SupplierOrganization` | 供应商组织(供货企业),N 条数据 |

### 4.2 为什么不共用一张 Organization 表

| 维度 | 共用一张表 | 双实体设计(已选) |
|---|---|---|
| 字段差异 | 字段塞在一起,有大量字段对其中一方无意义 | 各自字段独立,语义清晰 |
| 业务演进 | 任何一方加字段都影响另一方 | 互不干扰 |
| 命名清晰度 | `Organization` 含糊 | 一眼看懂边界 |
| 表查询 | 各处 `WHERE type='buyer'` 过滤 | 直接独立表查询 |

**判断逻辑**:Supplier 和 Buyer 业务属性差异巨大,共用一张表是过度抽象。

### 4.3 一企业多账号支持

```
SupplierOrganization  1 ──N  SupplierMember  N── 1  User
BuyerOrganization     1 ──N  BuyerMember     N── 1  User
```

**Member 中间表字段**:`id` / `user_id` / `org_id` / `is_owner` / `created_at`
- **不含 business_role 字段**(避免非标准做法,详见第五章 scope 设计)
- `is_owner` 标识企业管理员

### 4.4 MVP 阶段功能约束

- 供应商入驻时**自动创建** 1 条 SupplierMember(`is_owner=true`)
- 采购方账号创建时**自动关联**到中建三局的 BuyerOrganization
- **不暴露"邀请员工"功能**(V1 再做,届时不改 schema)

## 五、系统角色设计

### 5.1 4 个系统角色(MVP 锁定)

```
Role 表初始数据:
┌────┬────────────┬──────────────┬────────┐
│ id │ code       │ name         │ scope  │
├────┼────────────┼──────────────┼────────┤
│ 1  │ PURCHASER  │ 项目部采购员  │ GLOBAL │
│ 2  │ SUPPLIER   │ 供应商       │ GLOBAL │
│ 3  │ OPERATOR   │ 平台运营     │ GLOBAL │
│ 4  │ ADMIN      │ 系统管理员   │ GLOBAL │
└────┴────────────┴──────────────┴────────┘
```

### 5.2 4 个角色的本质区别

| 角色 | 业务定位 | 关注什么 | 是否触碰业务数据 |
|---|---|---|---|
| **PURCHASER** | 业务用户 - 业主侧 | 项目、采购、询价、订单 | ✅ 业务操作主体 |
| **SUPPLIER** | 业务用户 - 供货侧 | 上架、报价、履约 | ✅ 业务操作主体 |
| **OPERATOR** | 平台内部 - 业务运营 | 审核供应商/商品、维护国别数据、风控监控 | ✅ 审核与监控 |
| **ADMIN** | 平台内部 - 技术管理 | 用户管理、角色权限、系统配置 | ❌ 不应触碰业务数据 |

### 5.3 OPERATOR vs ADMIN 拆分的设计理由

这是标准的"业务管理员 vs 系统管理员"分离:

- 各大 B2B 平台("运营后台"和"系统后台"两套体系)的通用做法
- 安全合规角度:符合"职责分离"——能管系统的人不应该能直接操作业务,反之亦然
- 与可行性报告中"等保三级"的合规目标对齐

### 5.4 Role.scope 字段(Q23 待定)

`scope` 字段标识角色的**作用域**:

| scope 值 | 含义 | MVP 是否使用 |
|---|---|---|
| `GLOBAL` | 全局角色(系统级) | ✅ MVP 4 个角色全部为 GLOBAL |
| `BUYER_ORG` | 组织内角色(BuyerOrg 内细分) | 预留 |
| `SUPPLIER_ORG` | 组织内角色(SupplierOrg 内细分) | 预留 |

未来扩展示例:
```
Role 表新增数据(V1+):
┌────┬──────────────────┬────────────┬──────────┐
│ id │ code             │ scope      │ scope_id │
├────┼──────────────────┼────────────┼──────────┤
│ 5  │ BUYER_PURCHASER  │ BUYER_ORG  │ 1        │  ← 中建三局内的采购员
│ 6  │ BUYER_APPROVER   │ BUYER_ORG  │ 1        │  ← 中建三局内的审批官
└────┴──────────────────┴────────────┴──────────┘
```

## 六、scope 与 module 的区别 ⭐

这是个**常见混淆点**。两者看着都像"分类",实则完全不同。

### 6.1 一句话区分

| 字段 | 在哪张表 | 回答什么问题 | 影响业务逻辑吗 |
|---|---|---|---|
| **`scope`** | `Role` 表 | 这个**角色**的作用域有多大? | ✅ 影响 |
| **`module`** | `Permission` 表 | 这个**权限点**属于哪个业务模块? | ❌ 不影响 |

### 6.2 类比记忆

| 字段 | 类比 |
|---|---|
| `Role.scope` | 像**护照的签发国**——决定护照在哪些地方有效 |
| `Permission.module` | 像**图书的分类标签**——只是为了图书馆好找,不影响内容 |

### 6.3 UI 类比

假设有一个"角色权限管理后台"页面:

```
┌─────────────────────────────────────────────────────┐
│  编辑角色:供应商 (SUPPLIER)                          │
│  作用域: 全局 GLOBAL          ← 这是 Role.scope     │
├─────────────────────────────────────────────────────┤
│  分配权限点:                                          │
│                                                       │
│  ▼ 商品管理(module=product)   ← 这是 Permission.module│
│    ☑ 查看商品 (product:read)                          │
│    ☑ 创建商品 (product:create)                        │
│    ☐ 审核商品 (product:approve)                       │
│                                                       │
│  ▼ 询价管理(module=rfq)                              │
│    ☑ 响应询价 (rfq:respond)                           │
│    ☐ 发起询价 (rfq:create)                            │
│                                                       │
│  ▼ 订单管理(module=order)                            │
│    ☑ 履约打卡 (order:milestone:checkin)               │
└─────────────────────────────────────────────────────┘
```

- `scope='GLOBAL'` 在**角色头部**,描述这个角色本身的属性
- `module='product/rfq/order'` 在**权限点列表的分组标题**,帮助分类展示

### 6.4 常见错误表述

| 错误 ❌ | 正确 ✅ |
|---|---|
| "这个权限点的 scope 是 product" | "这个权限点的 module 是 product" |
| "这个角色的 module 是 GLOBAL" | "这个角色的 scope 是 GLOBAL" |

## 七、5 张标准 RBAC0 表

```
User              用户(登录身份)
Role              角色
Permission        权限点
UserRole          用户-角色关联(N:N)
RolePermission    角色-权限关联(N:N)
```

### 7.1 表结构概览

#### Role

| 字段 | 类型 | 说明 |
|---|---|---|
| id | PK | |
| code | string unique | 角色 Code |
| name | string | 显示名称 |
| scope | enum | GLOBAL / BUYER_ORG / SUPPLIER_ORG(见 Q23) |
| scope_id | int nullable | 组织内角色时关联的组织 ID |

#### Permission

| 字段 | 类型 | 说明 |
|---|---|---|
| id | PK | |
| code | string unique | 权限点 Code(如 `product:approve`) |
| name | string | 显示名称 |
| module | string | 所属业务模块(分类标签) |
| description | text | 权限点说明 |

#### UserRole / RolePermission

均为标准 N:N 关联表(`id` / 两个外键 / `created_at`)。

### 8.2 权限校验完整链路图

```
┌─────────────────────────────────────────────────────────┐
│ 张三登录系统                                              │
└──────────────────────────┬──────────────────────────────┘
                           ↓
                  查 UserRole 表
                  user_id=1 → role_id=1 (PURCHASER)
                           ↓
                  查 RolePermission 表
                  role_id=1 → permission_ids=[1, 4]
                           ↓
                  查 Permission 表
                  → codes=['product:read', 'rfq:create']
                           ↓
                  写入 JWT/Session,返回前端
                           ↓
┌─────────────────────────────────────────────────────────┐
│ 前端拿到 permissions=['product:read', 'rfq:create']      │
└──────────────────────────┬──────────────────────────────┘
                           ↓
              ┌────────────┴────────────┐
              ↓                         ↓
       【按钮渲染时】              【点击按钮发请求】
        hasPermission('rfq:create')   POST /api/rfq
              │                         │
              ├─ true → 显示按钮         ↓
              └─ false → 隐藏按钮    后端 API 守卫拦截
                                  @RequirePermission('rfq:create')
                                        │
                                  ┌─────┴─────┐
                                  ↓           ↓
                                包含权限      不包含
                                  ↓           ↓
                              放行业务      403 Forbidden
                              逻辑          (即便前端被绕过也拦住)
```

### 8.3 这个机制为什么"标准且优雅"

| 场景 | 操作 |
|---|---|
| 加新角色 | 在 Role 表加一行 + RolePermission 表加几行,**零代码改动** |
| 加新权限点 | 在 Permission 表加一行 + 给相关 Role 关联,**零代码改动** |
| 改角色权限 | 改 RolePermission 表,**零代码改动** |
| 代码侧 | 只关心"这个 API 要求 `xxx:xxx` 权限点",不关心谁有,**解耦** |
| 前后端一致 | 同一份权限点列表,前端做显隐,后端做校验,**规则统一** |

## 九、权限点颗粒度与实战示例

### 9.1 颗粒度选择:资源+动作级

格式:`resource:action`,如 `product:create`、`supplier:approve`

### 9.2 资源+动作级的实战示例

以 `product` 资源为例,4 角色的动作权限:

| 权限点 | PURCHASER | SUPPLIER | OPERATOR | ADMIN |
|---|---|---|---|---|
| `product:read` | ✅(看公开的) | ✅(看自己的) | ✅(看全部) | ❌ |
| `product:create` | ❌ | ✅ | ❌ | ❌ |
| `product:update` | ❌ | ✅(改自己的) | ✅(改任何) | ❌ |
| `product:delete` | ❌ | ✅(删自己的) | ✅ | ❌ |
| `product:approve` | ❌ | ❌ | ✅ | ❌ |
| `product:reject` | ❌ | ❌ | ✅ | ❌ |

### 9.3 关键洞察:权限校验 vs 数据权限解耦

**权限点只回答"能不能做这个动作",至于"能对哪些数据做"是数据权限的事。**

```
权限校验:    "你有没有资格做 product:update?" → 是 / 否
数据权限:    "你能改哪些 product?"            → WHERE supplier_id = currentUserId
```

#### 执行流程

```
API 入口
   ↓
权限校验 (product:update?)
   ├─ 无权 → 403
   └─ 有权 → 进入业务逻辑
              ↓
        数据查询时自动加 WHERE supplier_id = currentUserId
              ↓
         ┌────┴────┐
         ↓         ↓
      命中数据    没命中
         ↓         ↓
      操作成功   404 (假装不存在,不暴露存在性)
```

## 十、6 个粒度选择的最终决定

| 选择 | 决定 |
|---|---|
| 1. 权限点颗粒度 | **资源+动作级**(`resource:action`),不做接口级,不做字段级 |
| 2. 是否分模块 | ✅ Permission 表加 `module` 字段 |
| 3. 角色-权限关系定义方式 | 📌 **Q22 挂起**,见第十三节 |
| 4. UserRole 关系 | **N:N**(数据结构标准,业务层 MVP 强约束 1:1) |
| 5. 权限校验层次 | 后端 API + 前端路由 + 前端按钮**三级校验**;字段级不做 |
| 6. 命名规范 | **冒号分隔** `resource:action` |

## 十一、权限校验三级机制

```
┌─────────────────────────────────────────────────────┐
│ 1. 后端 API 守卫(必做,安全底线)                       │
│    每个 API 入口检查权限点                              │
│    @RequirePermission('supplier:approve')             │
├─────────────────────────────────────────────────────┤
│ 2. 前端路由守卫(必做)                                 │
│    进入页面前检查权限,无权直接跳转                      │
├─────────────────────────────────────────────────────┤
│ 3. 前端按钮级隐藏(必做)                               │
│    无权限的操作按钮不显示(UX 体验)                     │
└─────────────────────────────────────────────────────┘
```

### 实现约定(伪代码示意)

```
登录/会话接口返回:
{
  user: { ... },
  permissions: ['supplier:approve', 'product:create', ...]
}

前端按钮:
if (hasPermission('supplier:approve')) {
  显示"审核通过"按钮
}
```

### 安全底线

- 后端 API 永远做独立校验,**绝对不信任前端**
- 前端隐藏只是 UX,不是安全机制
- 字段级权限不做

## 十二、数据权限设计(归属过滤,非策略引擎)

### 12.1 核心原则

- **公开池**:任何登录用户可读
- **私有池**:按归属字段(`supplier_id` / `buyer_org_id`)过滤

### 12.2 数据权限矩阵(按实体维度)

| 数据实体 | PURCHASER | SUPPLIER | OPERATOR | ADMIN |
|---|---|---|---|---|
| 商品 (SKU) | 所有已上架的 | 仅自己上架的 | 全部(含未上架/驳回) | ❌ |
| 供应商档案 | 所有已通过审核的 | 仅自己的 | 全部 | ❌ |
| 国别准入数据 | 全部公开(只读) | 全部公开(只读) | 全部 + 可编辑 | ❌ |
| 项目 | 仅自己 BuyerOrg 的 | ❌ | 全部 | ❌ |
| 采购清单 | 仅自己 BuyerOrg 的 | ❌ | 全部 | ❌ |
| 询价单 (RFQ) | 仅自己 BuyerOrg 发起的 | 仅被分发给自己的 | 全部 | ❌ |
| 报价 | 自己 RFQ 收到的所有报价 | 仅自己提交的 | 全部 | ❌ |
| 订单 | 仅自己 BuyerOrg 的 | 仅自己参与的 | 全部 | ❌ |
| 会员/缴费记录 | ❌ | 仅自己的 | 全部 | ❌ |
| 风控驾驶舱 | 待定 Q11 | 待定 Q11 | ✅ | ❌ |
| 用户/角色/权限 | ❌ | ❌ | ❌ | ✅ 全部 |
| 系统配置/日志 | ❌ | ❌ | ❌ | ✅ 全部 |

### 12.3 提炼出的核心原则

1. **公开池 vs 私有池**
 - 公开池:已上架商品、已审核供应商、国别准入数据
 - 私有池:项目、清单、RFQ、报价、订单、会员记录

2. **私有池的归属规则**
 - SUPPLIER 私有池 → 按 `supplier_id`(账号即组织)
 - PURCHASER 私有池 → 按 `buyer_organization_id`(组织即边界)
 - OPERATOR → 无业务归属约束,全平台业务数据可见
 - ADMIN → 不访问业务数据,只能操作系统数据

3. **职责分离原则**
 - OPERATOR 操作业务但不动系统
 - ADMIN 操作系统但不动业务

## 十三、Seed 与「初始超级管理员模式」

### 13.1 名词解释

**「初始超级管理员模式」**(也叫"种子管理员模式"):一种业务设计套路,**不是技术栈**。

> 系统刚部署完,数据库是空的。需要一个用户才能创建用户,但需要先有用户才能登录创建——这是"鸡和蛋"问题。
>
> 解法:seed 脚本预先种 1 个超级管理员账号(我们称之为 super admin),作为"零号账号"。运维用它首次登录,创建后续正式账号。

类比:新装的路由器出厂默认有 `admin/admin`,登录后改密码、创建其他账号——是同一个模式。

### 13.2 Seed 写死范围

| 数据 | 是否脚本写死 | 说明 |
|---|---|---|
| Role 表(4 个角色定义) | ✅ | 角色种类固定 |
| Permission 表(权限点定义) | ✅ | 开发期确定 |
| RolePermission 关联 | ✅(见 Q22) | 业务约定,跟代码进 Git |
| 初始 super admin 账号 | ✅ 1 个 | 零号账号,引导用 |
| BuyerOrganization(中建三局) | ✅ 1 条 | MVP 唯一业主 |

### 13.3 Seed 不写死范围

| 数据 | 来源 |
|---|---|
| 后续 ADMIN 账号 | 由 super admin 在系统后台创建 |
| OPERATOR 账号 | 由 ADMIN 在系统后台创建 |
| PURCHASER 账号 | 由 BuyerOrg 的 owner 创建/邀请 |
| SUPPLIER 账号 | 自助注册 + OPERATOR 审核 |
| SupplierOrganization | 供应商入驻流程产生 |

### 13.4 引导流程图

```
[系统部署完成]
       ↓
[seed 脚本运行]
       ├─ 初始化 Role / Permission / RolePermission
       ├─ 创建 1 条 BuyerOrganization(中建三局)
       └─ 创建 1 个 super admin 账号(零号账号)
       ↓
[运维首次用 super admin 登录]
       ↓
[强制修改初始密码]
       ↓
[super admin 在系统后台创建正式账号]
       ├─ 创建若干 ADMIN
       ├─ 创建若干 OPERATOR
       └─ 创建若干 PURCHASER(关联中建三局)
       ↓
[正式运营]
       ├─ ADMIN 持续管理用户/系统
       ├─ OPERATOR 审核供应商入驻
       ├─ 供应商自助注册 → 经 OPERATOR 审核 → 获得 SUPPLIER 角色
       └─ super admin 可禁用或保留作为应急账号
```

### 13.5 为什么不能全部账号都用 seed

- 加运营人员要发版?不现实
- 平台运维和开发耦合
- 离职员工账号怎么处理?也要发版?

**结论**:seed 只解决"零号账号"问题,其他账号靠业务流程产生。

### Q22:角色-权限关系采用代码定义还是数据库定义?

#### 背景
讨论标准 RBAC0 的 RolePermission 表如何维护。我们已经定了 5 张表的结构(包括 RolePermission),但"角色和权限的关联数据"是放在数据库里(seed 维护)还是写在代码常量里,有争议。

#### 方案对比

| 维度 | A. 纯数据库定义 | B. 纯代码定义 | C. 配置文件 + 启动同步 |
|---|---|---|---|
| 表结构 | 标准 RolePermission 表 | 无关联表,代码里写映射 | 标准 RolePermission 表 |
| 权限定义来源 | 数据库(seed 初始化) | 代码常量/配置 | 代码配置文件(唯一可信来源) |
| 同步机制 | 人工 / seed 脚本 | 无 | 启动时比对配置与数据库,自动同步 |
| 是否符合标准 RBAC | ✅ 是 | ⚠️ 伪 RBAC | ✅ 是 |
| 权限变更操作 | 改 seed + 重跑 / 直接 SQL | 改代码 + 发版 | 改配置 + 重启 |
| Git 版本可追溯 | ⚠️ 运行时数据脱离 Git | ✅ 完全在 Git | ✅ 完全在 Git |
| 未来加权限管理后台 | 直接基于现有表加 UI | **需要重构 schema** | 直接基于现有表加 UI |
| MVP 工作量 | 中(写 seed) | 小(写常量) | 中(写配置 + 同步逻辑) |

#### 预倾向

**方案 C**——保留标准 RBAC 表结构 + 代码可追溯,折中最优。**B 不推荐**,因为意味着未来做后台时必须重构 schema。

#### 待与同事讨论的关键点

- 团队对"权限定义跟代码一起进 Git"的倾向?
- 是否能接受"启动同步"这种半自动机制?
- V1.0 阶段是否就要做权限管理后台?如果是 → 选 A;如果不是 → C 更优

### Q23:Role 表是否引入 scope 字段?

#### 背景
讨论"标准 RBAC 应不应该包含 business_role 字段"。结论是:不要在 Member 中间表加 business_role(非标准做法),要做组织内角色应该走 Role 表 + `scope` 字段(标准做法)。本待定点决定 MVP 是否预留这个字段。

#### 方案对比

| 方案 | 描述 |
|---|---|
| A. 不加 scope | MVP schema 更简,未来要做组织内角色时需改 schema |
| B. 加 scope 字段,MVP 只用 GLOBAL | 标准做法,零成本预留 |

#### 预倾向

**B**——零工作量,符合标准 RBAC,避免未来重构。

---

### Q24:OPERATOR 是否需要进一步细分?

#### 背景
在引入 4 角色(PURCHASER/SUPPLIER/OPERATOR/ADMIN)时,讨论 OPERATOR 内部是否要细分子角色,例如审核员、风控员、内容运营。

#### 方案对比

| 方案 | 描述 |
|---|---|
| A. MVP 不细分 | 所有 OPERATOR 共享一组权限 |
| B. MVP 细分子角色 | 如 `OPERATOR_AUDITOR`、`OPERATOR_RISK`、`OPERATOR_CONTENT` |

#### 预倾向

**A**——MVP 内部员工有限,细分扩大权限点矩阵复杂度。

---

### Q25:ADMIN 是否能访问业务数据?

#### 背景
在拆分 OPERATOR/ADMIN 时,讨论"职责分离"的严格程度。ADMIN 默认只管系统(用户、角色、配置),不碰业务(订单、RFQ、商品)。但当 ADMIN 需要排查业务问题时,是否给一个"临时切换为 OPERATOR 身份"的口子(称作 impersonation)?

#### 方案对比

| 方案 | 描述 |
|---|---|
| A. 严格分离 | ADMIN 完全看不到业务数据 |
| B. 允许 impersonation | ADMIN 可临时切换为 OPERATOR 身份排查问题 |

#### 预倾向

**MVP 默认 A**——严格分离,符合等保 2.0 三级。如果 ADMIN 真要排查业务,通过审计日志而非直接访问。

---

### Q26:初始 super admin 的密码策略?

#### 背景
「初始超级管理员模式」(第十三章)下,seed 脚本要创建"零号账号"。这个账号的密码怎么处理是个安全问题——写死在代码里,代码库泄露就完蛋。

#### 方案对比

| 方案 | 描述 | 风险 |
|---|---|---|
| A. seed 写死明文 | 开发期方便 | ⚠️ 生产期危险,代码库泄露 = 系统裸奔 |
| B. 启动时随机生成,输出日志 | 无代码泄露风险 | ⚠️ 运维要捞日志,流程繁琐 |
| C. 环境变量注入 | 与外部 secrets 管理工具配合 | ✅ 标准做法 |

#### 预倾向

**C** + 强制首次登录修改密码。

## 十五、已闭环的决策汇总

| 决策点 | 结论 |
|---|---|
| RBAC 复杂度档次 | 标准 RBAC0,认真做 |
| 两个核心维度 | 组织实体(2 个) vs 系统角色(4 个),独立配置 |
| 组织实体 | BuyerOrganization + SupplierOrganization(只此 2 个) |
| 用户-组织关系 | 通过 Member 中间表 N:N |
| Member 字段 | id / user_id / org_id / is_owner / created_at(**不含 business_role**) |
| MVP 暴露的账号能力 | 一企业一账号(数据结构支持多账号) |
| 系统角色 | PURCHASER / SUPPLIER / OPERATOR / ADMIN(4 个固定) |
| OPERATOR vs ADMIN 拆分 | 职责分离:业务管理 vs 系统管理 |
| 权限点颗粒度 | 资源+动作级 |
| Permission 是否分模块 | ✅ 加 module 字段 |
| UserRole 关系 | N:N(数据结构),1:1(MVP 业务约束) |
| 权限校验层 | 后端 API + 前端路由 + 前端按钮三级 |
| 字段级权限 | ❌ 不做 |
| 权限点命名规范 | `resource:action`(冒号分隔) |
| 数据权限实现 | 归属字段硬过滤(非策略引擎) |
| 数据权限矩阵 | 见第十二节 |
| 初始超级管理员模式 | seed 创建 1 个零号账号,后续账号通过系统后台创建 |
